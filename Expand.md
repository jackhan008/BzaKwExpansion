在BZA_keywords_expand文件夹下
实现一个品牌广告的拓词功能 用户输入一个search theme，系统会得到search theme 对应的SRPV 和 revenue

系统逻辑:
1. 输入一个 search theme，azure api 的 AI 根据这个search theme 判断这是品牌还是品牌具体的产品，如果是品牌 则 AI 可以扩展出这个品牌相关的 一些产品和品牌的搜索词；如果是产品 则AI可以扩展出这个产品相关的搜索词。扩展的要求是10个以内，一定要紧密相关且词根较少 当产品词独立的含义比较明确时,扩展词可以不用包括原来的品牌词, 扩展结果记录为[扩展词]
2. 将en-au-query.csv这个数据用sqlite和向量数据库存储(采用faiss存储) 由于这些query英文偏多 所以向量采用比较适用于英文的embedding 可以使用目前的sota embedding 模型本地运行得到，并且记住每一个query对应的srpv和click以及revenue
3. 遍历[扩展词]去搜索query列表，采用两种搜索方式 : a. 在数据库中采用前或后包含的硬匹配，(需要考虑query和扩展词都要去掉空格来匹配) 这样匹配到的query我们给他计分为2 并且计算相关性(按照词语的同现/max(querylen, 扩展词len)); b. 使用向量匹配的方式来匹配相关性高于一定阈值的query 这里可以匹配top100的query(且需要保障相关性高于0.8),这样匹配到的query还需要满足包含扩展词(需要去掉空格后匹配) 或者 去掉空格后编辑距离小于1/5(以满足typo), 这种匹配到的query给计分为1 .这样得到两个set的query, 再汇总得到最终的query set，并将计分加总。
4. 以上得到的query set 按照计分倒序排序，得到 query, 相关性, srpv, click, revenue这样的列表， 帮展示出来。
5. 用户不仅仅可以输入一个search theme 也可以输出一个list 我们要对这个list 都做这样的处理，最终得到一个汇总的文件可供下载。
6. 匹配出来的query，需要经过AI校验确保无误，校验方式如下：

query 不是一个新的品牌，不是因为typo导致的错拼；
不能是两个品牌的对比这样的逻辑，如果除去本品牌词后剩余的词还是一个品牌，则过滤；
如果原始品牌是通用词，匹配出来的query 如果有非品牌目的，也不能要。比如说aami 匹配出来aami park，说明用户主要目标不是想搜索aami，而是想去这个park
校验结果不是直接删除，而是输出时打上一个标签，说明该query AI校验后不通过

AI 拓词时，需要考虑以下几点：

如果该品牌名称都是日常用的词汇，比如compare/market/ink等，那扩展出来的词一定要包含品牌名称，可以是品牌名称里面嵌入词汇，但品牌含义不能改变
不能向上扩展，比如从A扩展出B，但B不仅仅是A，还有其他产品，这样就不行。例如gmail不能扩展出google workspace
